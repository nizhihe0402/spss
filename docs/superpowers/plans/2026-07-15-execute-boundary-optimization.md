# 后期优化：精准解析方案（EXECUTE 边界）

日期：2026-07-15
状态：**已确认方向，暂不实施**（用户决定记录待后期优化）

## 背景

当前规则块边界靠 `findNextHardBoundary` 的关键字列表（FREQUENCIES / VALUE LABELS /
SORT CASES / STRING / NUMERIC 等）截断——本质是"枚举所有非规则语句"的容错做法。
语义上更正确的边界是 SPSS 的 `EXECUTE.`（transformation 的强制执行点，天然的代码段
分隔符）。

## 语料实测（2026-07-15）

各 .sps 中以 EXECUTE 结尾的代码段占比：

| 脚本 | 代码段 | EXECUTE 覆盖 |
|------|--------|--------------|
| 表2-1 | 147 | 94% |
| 表2-2 | 124 | 97% |
| 表1-1 | 34 | 97% |
| 表1-3 | 10 | **70%（最差）** |
| 其余 | — | 92~93% |

**结论：不能只认 EXECUTE**——疾控人员手写脚本总有百分之几漏写，
漏写段落会吞掉后续语句导致解析错误。

## 方案（三步演进）

1. **短期（现状）**：关键字边界列表继续工作，作为不规范脚本的容错。
2. **中期**：`EXECUTE.` 提升为**首选边界**，关键字列表降级为漏写时的兜底。
   两者并存：规范脚本走语义正确路径，存量脚本不受影响。
   改动点：`RuleParser.findNextHardBoundary` / `nextRuleStart` /
   `findBlockEndForTarget` 优先取下一个 `EXECUTE.` 位置。
3. **配套 lint**：上传/reparse 时检测"未以 EXECUTE 结尾的代码段"，
   在解析结果中输出警告（行号+提示），引导脚本作者补全。
   用警告而非硬报错，兼容存量脚本。

## 验收要点（实施时）

- 全语料 10 个 .sps 解析结果与现状对比：规则数、目标、步骤数不回退；
- 表1-3（覆盖率最低）重点回归；
- 新增测试：漏写 EXECUTE 的块回落到关键字边界；
- lint 警告在表1-3 上应报出 3 处。
