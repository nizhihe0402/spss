# DO IF 块聚合为单条规则 — 设计文档

日期：2026-07-15
状态：已与用户确认

## 背景

表2-1.sps 的「身份证出生日期异常」块（约 265-295 行）当前被解析为 5-6 条碎片规则
（R004-R008、R040）：中间变量 SFZ_DATE / ID_DATE / BIRTH_DATE 各自成为独立规则，
且因源变量含 SFZ 被 BlockClassifier 误分类；同名目标（身份证出生日期异常）分散在
多条规则中。用户期望：这是**一组校验**，应合并为一条规则。

## 目标

每个顶层 `DO IF ... END IF` 块内的语句聚合为按序执行的规则步骤（steps），
块内中间变量不再产生独立规则。**通用机制**，对所有 .sps 生效（用户选择，
非仅针对单一块的特例）。

## 核心设计

### 1. 按「汇变量」拆分（不是按块单条）

一个顶层 DO IF 块产出 **N 条规则（N = 汇变量数）**：

- **汇（sink）** = 块内被赋值、但不被块内其他步骤消费的变量；
- 每条规则 = 汇变量 + 其块内依赖闭包步骤（被其直接/间接消费的中间变量计算），
  步骤按源码顺序排列；
- 同一块内同目标的多条碎片规则（如 IF 赋值 + COMPUTE）合并进同一条；
- 共享中间步骤复制到各汇的规则中；
- 块前紧邻的 `COMPUTE 汇 = $SYSMIS.` 初始化并入该规则。

规则属性：`target`=汇变量；`type`=BlockClassifier 按整块分类；
`sourceVariables`=各步骤源变量 − 块内中间目标；`spssSource`=整块文本（含 init）；
javaPreview / executionChain 走既有 steps 构建逻辑。

### 2. 附带修复：嵌套 DO IF 条件丢失

`findActiveDoIfCondition` 注释声称嵌套时 AND 链接，实际只返回栈顶——
`SFZ_DATE` 的条件是 `NOT(MISSING(SFZ)...)`，丢失了外层 `ZJTYPE = 1`。
改为整栈 AND 连接；多条件时各元素加括号（`(ZJTYPE = 1) AND (NOT(...))`），
避免 OR 优先级错误。单元素时保持原样不加括号（最小化 diff）。
ConditionExpression 已支持括号布尔分组（2026-07-15 修复）。

### 3. 行为变化点

- 块内常量赋值（如 `身份证出生日期异常 = 1`）当前被 `isConstantAssignment`
  全局跳过——合并块内**必须保留**为 step，否则分支逻辑不完整。
- 中间变量仍写入 RowContext，块外引用不受影响；需确认 AvailabilityChecker
  将合并规则的 step targets 计入「该规则提供的变量」。

## 全语料影响验证（已完成，按块位置精确扫描）

- **不受影响**：表1-2/1-3、表2-3、表3-x 全部；身高/体重块（39×2 个单目标小块，
  现有解析已按同目标聚合为各 1 条规则，聚合 pass 为恒等变换）；年龄异常/可疑、
  证件号码/位数块（单目标）；MatchSequence 块；DatasetRule/OutputRule 通道。
- **正确合并**：身份证出生日期异常块（单汇 → 1 条）；身份证性别异常块
  （#gender_bit 被消费，单汇 → 1 条）×2 脚本。
- **多汇块保持独立**（9 个多目标块全部验证）：教室年级/班级代码块（6 汇 → 6 条，
  与现状一致）；血压块 收缩压可疑/舒张压可疑/压差可疑（3 汇 → 3 条，与现状一致）。

## 测试计划（TDD）

1. 嵌套条件：内层 step 条件为 `(外层) AND (内层)` 形态；
2. 出生日期块 → 恰好 1 条规则，target/type/steps 顺序/每步完整条件/
   sourceVariables 断言；SFZ_DATE 等不再独立成规则；
3. 多汇块（模拟血压块）→ 保持 N 条独立规则；
4. 执行测试：RuleExecutor 跑合并规则（日期一致→0；不一致→1；SFZ 缺失→1）；
5. 全语料回归：所有 .sps 解析不抛错，身高/体重等不受影响块规则数不变；
   现有测试套件全绿。
